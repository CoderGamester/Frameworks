# Architecture Log

## 2026-05-12 — MobileServices simulator: per-target routing + UX polish (still v1.0.0 unreleased)

### Architectural Decisions
- Replaced the previous "single global broker + bidirectional authority handoff" design (`MobileSimulatorState.IsActivePluginConnected` / `PluginConnectedChanged` greying out the Explorer dropdown when the Device Simulator plugin attaches) with **per-target routing**: a new `[Flags] enum SimulatorTarget { None, StandaloneWindow, RuntimeOverlay, All }` carried by every `Push*`, default `All`. Each renderer filters at handler entry on its own bit. Resolves the two stacked complaints "Explorer dropdown is greyed when the plugin is alive" and "plugin clicks paint into the standalone window" with one mechanism — neither symptom can recur because the surfaces no longer share state.
- Split `MobileSimulatorState.Platform` into per-surface `WindowPlatform` (driven by the Explorer dropdown) and `OverlayPlatform` (auto-synced by the Device Simulator plugin from `Application.platform`), each persisted to its own `EditorPrefs` key. Eliminates the "two producers competing for one variable" race that the bidirectional-authority signal was working around.
- Codified an **editor-only opt-in surface visibility pattern** (`MobileServicesDeviceSimulatorPlugin.BuildOverlayHintBanner` + `RefreshOverlayHintBanner`): when a UI panel's "happy path" depends on a runtime opt-in setting AND a runtime state (e.g. `EnableRuntimeSimulatorOverlay = true` AND play mode), surface the missing precondition via an inline auto-hiding banner driven by an existing periodic poll, rather than letting the user discover the silent no-op. Banner state machine: setting OFF → "enable + Open Project Settings →" CTA; setting ON in edit mode → "press Play"; setting ON in play mode → hidden. Same 500 ms `schedule.Execute(...).Every(500)` poll already used for `SyncPlatformFromHost` drives the refresh.
- Codified an **iOS/Android UI parity discoverability pattern**: when a button represents a feature with no equivalent on one platform (e.g. iOS action sheet collapses to the same Material 3 dialog as the modal alert on Android), grey it out (via `SetEnabled(false)`) on the unaffected platform and surface a tooltip explaining the OS-level cause. Discoverable in 200 ms peripheral vision rather than buried in source comments. Applied to the `Show Action Sheet` button in `NativeUiTab` reactively to `WindowPlatform`.

### Structural Changes
- `Packages/com.gamelovers.mobileservices/Editor/Explorer/Overlays/MobileSimulatorState.cs`: rewritten — added `SimulatorTarget`, split `Platform` into `WindowPlatform` / `OverlayPlatform` with separate `EditorPrefs` keys + separate `*PlatformChanged` events, removed `IsActivePluginConnected` / `PluginConnectedChanged` / `_isPluginConnected`, every `Push*` gained `SimulatorTarget targets = SimulatorTarget.All`, every `*Requested` event signature became `Action<SimulatorTarget, TSpec>`.
- `MobileSimulatorWindow.cs` / `MobileSimulatorRuntimeOverlay.cs`: each gained a `TargetsThisSurface(SimulatorTarget)` filter at the top of every `On*` payload handler; bound to their own `WindowPlatformChanged` / `OverlayPlatformChanged` event respectively.
- `MobileServicesExplorerWindow.cs`: dropped `OnPluginConnectedChanged` / `ApplyPluginConnectedState` plumbing (dropdown stays interactive forever), dropdown writes to `WindowPlatform`, added inline italic scope-hint label + tooltip ("Controls the Mobile Simulator window only.") with a new `.explorer-header-hint` USS rule.
- `MobileServicesDeviceSimulatorPlugin.cs`: every push call passes `SimulatorTarget.RuntimeOverlay` (constant `OverlayOnly`); `SyncPlatformFromHost` writes `OverlayPlatform` only; `OnCreate` no longer toggles `IsActivePluginConnected`, `OnDestroy` removed; `Dismiss all mocks` button switched from `EditorPlatformSimulator.DismissAllOverlays()` (broadcast) to `MobileSimulatorState.PushDismissAll(OverlayOnly)`. New `BuildOverlayHintBanner` / `RefreshOverlayHintBanner` + `.msp-overlay-hint` / `.msp-overlay-hint-label` USS.
- `NativeUiTab.cs`: new `MakePrimaryButtonRow(params Button[])` helper wrapping primary buttons so they shrink-wrap to natural width (matches Alerts pair) — applied to Toast / Review / Share + the Alert pair refactored through it for consistency. Toast section's `Long duration` toggle text now interpolates `(3.5s vs 2.0s)` from new central `ToastShortSeconds` / `ToastLongSeconds` constants and carries a tooltip mapping to Android's `Toast.LENGTH_SHORT/LENGTH_LONG` (iOS fakes the toast and reuses the same timings). New `_actionSheetBtn` field + `OnWindowPlatformChanged` / `ApplyActionSheetButtonState` reactively grey out the action sheet button on Android with an OS-contract tooltip.
- `Packages/com.gamelovers.mobileservices/AGENTS.md`: §2 Editor block rewritten — Explorer dropdown paragraph spells the per-surface scope; "Truth-mirror simulator — three rendering surfaces" header replaced with "two renderer surfaces, one broker" + per-target routing paragraph; plugin paragraph updated to "scoped to RuntimeOverlay" + new precondition-banner sentence.

- `Packages/com.gamelovers.mobileservices/CHANGELOG.md`: folded into existing `## [1.0.0]` section per pre-publication versioning rule. ### Data Flows
- **Per-target broker push** (NEW): producer calls `MobileSimulatorState.PushX(spec, SimulatorTarget.X | Y)` → broker invokes `XRequested` with the target mask + spec → every alive subscriber's `OnX(targets, spec)` runs → first line `if (!TargetsThisSurface(targets)) return;` short-circuits non-matching renderers. Default `All` keeps Explorer-tab and `EditorPlatformSimulator` call sites unchanged. Plugin scopes its pushes to `RuntimeOverlay` only.
- **Per-surface platform sync** (NEW): Explorer dropdown → `MobileSimulatorState.WindowPlatform = X` → `WindowPlatformChanged(X)` → `MobileSimulatorWindow` re-skins. In parallel: plugin's 500 ms poll reads `Application.platform` (spoofed by Unity's Device Simulator) → `MobileSimulatorState.OverlayPlatform = X` → `OverlayPlatformChanged(X)` → `MobileSimulatorRuntimeOverlay` re-skins. The two flows never cross — one variable per surface, one event per surface.
- **Reactive action-sheet grey-out** (NEW): `NativeUiTab.BuildUi` subscribes to `MobileSimulatorState.WindowPlatformChanged`; `RegisterCallback<DetachFromPanelEvent>` unsubscribes when the tab leaves the panel; `ApplyActionSheetButtonState(SimulatedPlatform)` flips `_actionSheetBtn.SetEnabled(!isAndroid)` + sets/clears the OS-contract tooltip. Initial state applied right after construction so the grey-out is correct on first open.
- **Auto-hiding precondition banner** (NEW): plugin's existing 500 ms `schedule.Execute(...).Every(500)` poll now drives both `SyncPlatformFromHost` AND `RefreshOverlayHintBanner`. Banner reads `MobileServicesSettings.instance.EnableRuntimeSimulatorOverlay` + `EditorApplication.isPlayingOrWillChangePlaymode`; both true → hidden, otherwise text + (optional) `Open Project Settings →` jump button.

## 2026-05-11 — MobileServices in-Game-view simulator (still v1.0.0 unreleased)

### Architectural Decisions
- Adopted **one broker, three rendering surfaces** for the Mobile Services truth-mirror simulator: the existing `MobileSimulatorState` singleton remains the single fan-out, now feeding (1) the standalone `MobileSimulatorWindow` (edit + play mode, unchanged), (2) the new `MobileSimulatorRuntimeOverlay` editor-only `[InitializeOnLoad]` bootstrap that spawns a `[EditorOnly] MobileSimulatorOverlay` `DontDestroyOnLoad` GameObject with a `UIDocument` rendered inside Unity's Game / Simulator viewport at the simulated-device pixel grid (play-mode-only, opt-in via `MobileServicesSettings.EnableRuntimeSimulatorOverlay`), and (3) the new `MobileServicesDeviceSimulatorPlugin` `UnityEditor.DeviceSimulation.DeviceSimulatorPlugin` subclass embedded inside Unity's Device Simulator Control Panel (auto-discovered, Simulator-view-only, no opt-in).
- Codified the **no-editor-tooling-tests** policy for `com.gamelovers.mobileservices`: all 5 existing editor-tooling test classes (`EditorPlatformSimulatorTest`, `MobileServicesBuildPostprocessorTest`, `MobileServicesExplorerWindowTest`, `MobileServicesSettingsTest`, `MobileSimulatorWindowTest`) + the `GameLovers.MobileServices.Editor.Tests` asmdef + `Editor/AssemblyInfo.cs` bridge file were removed. Rationale: those tests were verifying UIToolkit visual-tree wiring / EditorWindow lifecycle quirks rather than behaviour that affects players, and the maintenance cost exceeded the regression-catch value. Policy codified in `Tests/AGENTS.md` §1 (new "Editor tooling (NOT tested)" group), §9 (new coverage-gap entry explaining the v1.0.0 removal), §11 (new update trigger), and parent `AGENTS.md` §3 (Tests bullet rewritten + `InternalsVisibleTo` paragraph trimmed). Aligns the package with what `Tests/AGENTS.md` §10's layout table already implied — `EditMode/Editor/` was never documented.
- Adopted **bidirectional authority signal**: when the `DeviceSimulatorPlugin` is alive inside Unity's Simulator view, it flips a new `MobileSimulatorState.IsActivePluginConnected` flag to `true`, firing a `PluginConnectedChanged` event. The Explorer window subscribes and greys out (+ tooltips) its own `Render as: iOS | Android` dropdown — the platform-skin source-of-truth is now the Simulator view's device profile, and the dropdown becomes redundant. Plugin auto-syncs platform from `Application.platform` on a 500 ms `VisualElement.schedule.Execute(...).Every(...)` poll (chosen over `DeviceSimulator.deviceChanged` because the event's delegate signature is documented inconsistently across Unity 6 minor versions).

Functionally identical to the asset path the plan described, but editor-only by construction — there's no asset file consumers could accidentally reference from a runtime `UIDocument`. Documented as an in-line deviation from the approved plan.
- Established the **opt-in default-OFF posture** for the runtime overlay: spawning a `DontDestroyOnLoad` GameObject in every consumer's play scene is opt-in, gated on `MobileServicesSettings.EnableRuntimeSimulatorOverlay` (default `false`). Surfaced in Project Settings under a new "Editor tooling" section. Lifecycle is strict: spawn on `EnteredPlayMode`, destroy instantly on `ExitingPlayMode` — no paused-snapshot preservation.

## 2026-05-11 — MobileServices DX overhaul (still v1.0.0 unreleased — 5 phases A→B→D→C→E)

### Architectural Decisions
- Adopted the **truth-mirror-window-instead-of-Game-View-overlay** pattern for the new `MobileSimulatorWindow`: positioned explicitly as **complementary** to Unity's built-in Device Simulator (`Window > General > Device Simulator`), not a replacement. Unity covers device frame + safe-area + `Application.platform` spoofing; the Mobile Simulator paints native OS chrome (alerts, action sheets, toasts, share sheets, review prompts, permission dialogs, ATT dialogs, heads-up notification banners) that lives in the OS process and Unity has no way to render. Acknowledged three overlap points (`SetSafeArea`, `SetIosLowPowerMode`, platform-skin toggle) and justified them on the programmatic / test-driven side that Unity's interactive-only simulator can't serve. Documented in `Packages/com.gamelovers.mobileservices/docs/explorer.md` `## Comparison with Unity's Device Simulator`.
- Standardised the **XML doc verbosity framework** for new code in the GameLovers package family: verbose `<summary>` + `<remarks>` blocks belong only on public consumer-facing interfaces (IntelliSense discoverability); `internal` types, editor-only types, and tab/window classes get one-sentence summaries that cross-reference `AGENTS.md` or `docs/`. Drift control — architecture rationale lives in `AGENTS.md` (contributors) and `docs/` (consumers), never duplicated in class headers. Applied in this session as a trim pass across 22 files (~145 LoC of XML doc removed).
- Codified the `EditorWindow` vs `VisualElement` `schedule` divergence: `EditorWindow` does not expose `schedule`; routing deferred work from a window uses `rootVisualElement.schedule.Execute(...)`. `VisualElement`-derived editor types (e.g. `MobileServiceTab`) inherit `schedule` directly. Common compile error: `CS0103: The name 'schedule' does not exist in the current context`. Surfaced as 2 errors in `MobileSimulatorWindow.cs` during the iOS-batch verification pass.
- Established the **fail-build-by-default + soft-mode placeholder** pattern for iOS submission-blocking config: `MobileServicesBuildPostprocessor` throws `BuildFailedException` listing every missing `NS*UsageDescription` key by default; soft mode (`AllowPlaceholderUsageDescriptions = true`) injects `[GameLovers placeholder]` strings that Apple will reject on submission — by design, to make accidental ship impossible. Pairs with a `Project Settings > GameLovers > Mobile Services` UIToolkit panel rendering each key with a `Suggest copy` button, capability toggles, project-scan auto-fill, and an iOS Privacy Nutrition Label draft generator.
- Adopted **code-only samples** (runtime-built `UnityEngine.UI` canvas, zero `.unity`/`.prefab`/`.meta` files) for the four mobileservices `Samples~/`. Deliberate divergence from the user's stated preference for hand-authored prefabs (`#12` in `.personal-memory.md`) — accepted in-session for the no-asset-dependency / diff-friendly trade-off but **flagged as follow-up: migrate to prefabs before tagging v1.0.0**. Sample-only types live in `GameLovers.MobileServices.Samples.<Name>` namespaces and are NOT public API surface.
- Centralised the **`HapticEnvelopes` table** (`Runtime/Haptics/Internal/HapticEnvelopes.cs`, always-compiled `internal static`) as the single source of truth for the per-preset `(timings_ms, amplitudes_0..255)` data feeding both `AndroidHapticsBackend` at runtime AND the Mobile Services Explorer's haptic envelope graph. Eliminates the editor-preview-vs-device-playback drift risk.
- Established **editor-only `internal static` override hooks** on host-driven services as the test/simulator wedge: `BatteryService.EditorLowPowerModeOverride` + `SimulateLowPowerModeChanged()`, `ConnectivityService.EditorReachabilityOverride` + `SimulateStatusChanged()`, `SafeAreaService.EditorSafeAreaOverride` + `SimulateSafeAreaChanged()`, `DeepLinkService.SimulateLinkActivated(Uri)`, `PermissionsService.EditorCheckOverride` / `EditorRequestOverride`, `AttService.EditorCurrentStatusOverride` / `EditorRequestResultOverride`. All `#if UNITY_EDITOR`-gated so player builds carry none of this surface. `EditorPlatformSimulator` static API is the public façade.

### Data Flows
- **Editor truth-mirror payload stream** (new): Explorer tab button → `MobileSimulatorState.Push*(SimulatedXSpec)` → state singleton fans payload to subscribed window → `MobileSimulatorWindow.OnX` clears stage + calls `MockBuilders.BuildX(platform, spec, dismiss)` → mounted `VisualElement` styled by the active platform USS (`MobileSimulator.iOS.uss` or `MobileSimulator.Android.uss`, swapped on the Explorer's `Render as` dropdown). - **`EditorPlatformSimulator` test-drive flow** (new): test/Explorer calls `EditorPlatformSimulator.SetIosLowPowerMode(true, batteryService)` → sets `BatteryService.EditorLowPowerModeOverride = true` (static, `#if UNITY_EDITOR`) → calls `batteryService.SimulateLowPowerModeChanged()` → runs `RefreshLowPowerMode()` → `QueryLowPowerMode()` reads the override → `OnLowPowerModeChanged` event fires for all subscribers. Same pattern across `SafeArea` / `Connectivity` / `DeepLink`. Permissions / ATT use static `Func<>` overrides consulted by the editor short-circuit branch of `RequestAsync` / `Check` / `CurrentStatus`.
- **iOS build-postprocessor mutation path** (new): `IPostprocessBuildWithReport.OnPostprocessBuild` runs on iOS → reads `MobileServicesSettings.instance` + `MobileServicesScanner.Scan()` → cross-references required vs configured → either throws `BuildFailedException` listing every missing key (default) or injects `[GameLovers placeholder]` strings (soft mode). Idempotent. Android path patches `Assets/Plugins/Android/mainTemplate.xml` with `<uses-permission>` entries + `<queries>` block.

URI shape: host treated as first path segment (`myapp://promo/abc` → `["promo", "abc"]`) so `/promo/:id` matches both bare-host and path-rooted forms.

Encoded as `AddressableIdsGeneratorUtils.ComputeFreshness` (cheap) + `Diff` (expensive) with a `FreshnessResult`/`DiffResult` mirror of the existing `GenerationResult` pattern. Rejected the obvious "live-diff every tick" option and the `AddressableAssetSettings.OnModification` event-subscription option — both add complexity disproportionate to value when an mtime-vs-mtime check answers "is the file stale?" in microseconds.
- Persisted last-generation snapshot in the existing `AddressableIdsEditorSettings` `ScriptableSingleton` rather than a sidecar file (`Library/AddressableIdsSnapshot.json`). Reasons: project-shared via VCS so all contributors see the same diff baseline; no separate lifecycle file to maintain; survives domain reload; atomic with the settings the snapshot was generated against (`_lastGenerationFilenameUsed` / `_lastGenerationLabelFilterUsed` ride along so the diff can flag "settings changed since last gen"). `RecordGeneration(addresses, labels)` is the only write path, called once at the end of `AddressableIdsGeneratorUtils.Generate` after `AssetDatabase.Refresh()`.
- Snapshot stored as ordinally pre-sorted `string[]` enables O(n+m) sorted set-diff on the read path (`SortedSetDiff` private helper) — no per-tick re-sorting, no `HashSet` allocation per diff. Sort happens once at write time.
- Preserved a pre-existing quirk in `GenerateAddressEnums` rather than fixing it as part of the refactor: the `name_filetype` collision-fallback updates the `addedNames` tracking list but the StringBuilder still emits the un-suffixed `GetCleanName(...)` for every entry. This means duplicate sanitized names produce CS0102 in the generated enum at consumer compile time. Extracted `ResolveSanitizedEnumName` as a shared helper used by both `GenerateAddressEnums` (preserves quirk) and the new `DetectSanitizedNameCollisions` (used for warnings) so a future fix can be made in one place — but flagged it explicitly as a "do not silently change" item in `AGENTS.md` §4 and in this log.
